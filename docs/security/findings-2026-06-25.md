## Finding 1 — Access token stored in `localStorage`

| Field | Detail |
|-------|--------|
| **Severity** | Medium |
| **File** | `core/ui/console/lib/auth.ts` |
| **Status** | Fixed |

### What was wrong

`localStorage.setItem(TOKEN_KEY, data.access_token)` persisted the console
admin JWT (scope `console:admin`) across browser restarts. Any XSS payload on
`console.sz3yan.com` could read it via `localStorage.getItem(...)` and exfiltrate
a token that grants full control over Mainframe, Janus, Aegis, Coder, and
Sentinel.

### Fix applied

Moved storage to `sessionStorage`. Token is now cleared when the browser tab
closes. Same-origin XSS can still read it within the session, but:

- Cross-tab exfiltration is eliminated.
- Persistence across restarts is eliminated.
- Scope also expanded to include `sentinel:read` so Sentinel BFF calls work.

```diff
- localStorage.setItem(TOKEN_KEY, data.access_token);
+ sessionStorage.setItem(TOKEN_KEY, data.access_token);
```

### Remaining risk

`sessionStorage` is still readable by same-origin JS. The correct long-term fix
is a server-side session (Mainframe already has `mf_sid` httpOnly cookies).
Wiring the console to use cookie-based sessions would eliminate the JS-readable
token entirely, but requires a server component or BFF refresh endpoint.
Track as a follow-up.

## Finding 2 — No token expiry check in `getToken()`

| Field | Detail |
|-------|--------|
| **Severity** | Medium |
| **File** | `core/ui/console/lib/auth.ts` |
| **Status** | Fixed |

### What was wrong

`getToken()` returned whatever string was in storage with no `exp` validation.
An expired access token would cause every API call to silently fail with a 401
redirect loop. There was also no way to proactively detect expiry before a
request went out.

### Fix applied

Added `jwtExp()` helper — decodes the JWT payload (base64url, no signature
verification needed here since we only care about the `exp` claim for UX
purposes; Mainframe validates signatures server-side). `getToken()` now:

1. Parses `exp` from the payload.
2. Compares against `Date.now() / 1000`.
3. If expired: removes the token from sessionStorage and returns `null`,
   triggering a re-login redirect at the next API call.

```typescript
function jwtExp(token: string): number | null {
  try {
    const payload = token.split(".")[1];
    if (!payload) return null;
    const json = JSON.parse(atob(payload.replace(/-/g, "+").replace(/_/g, "/")));
    return typeof json.exp === "number" ? json.exp : null;
  } catch { return null; }
}
```

## Finding 3 — Destructive operations use `confirm()`, no step-up auth

| Field | Detail |
|-------|--------|
| **Severity** | Low–Medium |
| **Files** | `middleware/stepup.py`, `KeysApp.tsx`, `CertsApp.tsx`, new `StepUpModal.tsx`, `lib/stepup.ts` |
| **Status** | Fixed |

### What was wrong

Key destroy (`DELETE /admin/kms/keys/{name}`) and cert revoke
(`POST /admin/pki/certificates/{serial}/revoke`) used `confirm()` as the only
confirmation gate. The browser `confirm()` dialog is suppressible by automation,
bypassable via XSS, and creates no audit trail.

Mainframe had a complete step-up auth subsystem (`middleware/stepup.py`,
`api/v1/stepup.py`) — single-use, Redis-backed, WebAuthn-verified tokens bound
to exact method+path — that was not wired to these routes.

### Fix applied

**`middleware/stepup.py`** — added two entries to `STEP_UP_REQUIRED`:
```python
r"DELETE /admin/kms/keys/.+":                 300,  # key destroy
r"POST /admin/pki/certificates/\d+/revoke":   300,  # cert revoke
```

The middleware now blocks those routes with `HTTP 401 WWW-Authenticate: step-up`
unless a valid single-use step-up JWT bound to that exact method+path is present
in `X-Step-Up: Bearer <jwt>`.

**`lib/stepup.ts`** (new) — full WebAuthn assertion flow:
1. `POST /auth/step-up/begin` → server challenge
2. `navigator.credentials.get({ publicKey: ... })` → passkey assertion
3. `POST /auth/step-up/complete` → step-up JWT (single-use, bound to method+path)

**`components/StepUpModal.tsx`** (new) — Windows 95-style modal overlay that:
- Auto-triggers the passkey prompt on mount
- Shows error + retry on failure
- Calls `onSuccess(jwt)` / `onCancel()` callbacks

**`KeysApp.tsx`** — Destroy button now sets `pending` state → shows `StepUpModal`
→ on success calls `destroyKey(name, stepUpJwt)` with the bound token.

**`CertsApp.tsx`** — same pattern for Revoke.

**`lib/api.ts`** — `destroyKey(name, stepUpToken)` and `revokeCert(serial, stepUpToken)`
now accept the step-up JWT; `req()` adds `X-Step-Up: Bearer <jwt>` header.

## Finding 4 — Socket proxy `POST=1` exposed to all containers on `orchubi_network`

| Field | Detail |
|-------|--------|
| **Severity** | Low–Medium |
| **File** | `docker-compose.edge.yml`, `docker-compose.sentinel.yml` |
| **Status** | Fixed |

### What was wrong

A single `szejo-control-plane-socket-proxy` with `POST=1` sat on `orchubi_network`.
Every container on that network (Mainframe, Janus, Aegis, Sentinel, Coder, etc.)
could reach it and issue POST requests to the Docker daemon. A compromised
container could create, start, or exec into other containers.

The sentinel-worker legitimately needs Docker access to run ephemeral scan
containers, but it consumed the same proxy as Watchtower, which had full
network/volume visibility.

### Fix applied

Split into two socket proxies:

**`szejo-control-plane-socket-proxy`** (unchanged) — on `orchubi_network`, full
perms for Watchtower (`CONTAINERS`, `EVENTS`, `INFO`, `IMAGES`, `NETWORKS`,
`VOLUMES`, `POST`).

**`szejo-control-plane-scan-proxy`** (new) — on the new isolated
`scan-proxy-network` (`szejo-scan-proxy-network`, `internal: true`). Only
`CONTAINERS`, `IMAGES`, `POST`. Only the sentinel-worker joins this network.

```
orchubi_network:
  socket-proxy (full) ← Watchtower

scan-proxy-network (internal):
  scan-proxy (CONTAINERS+IMAGES+POST only) ← sentinel-worker
```

Sentinel-worker was removed from `orchubi_network` entirely. It now uses:
- `sentinel-network` → DB + Redis
- `scan-proxy-network` → Docker socket (scoped)

## Finding 5 — Terraform state is an unprotected local file

| Field | Detail |
|-------|--------|
| **Severity** | Operational risk |
| **File** | `manifest/terraform/terraform.tfstate` |
| **Status** | Documented, example provided |

### What was wrong

`terraform.tfstate` is gitignored and stored only on the host. It contains the
live mapping between Terraform resource IDs and Cloudflare DNS records, tunnel
routes, and firewall rules. A host failure with no backup = losing all infra
config. Recovery via `import-existing.sh` is possible but requires working
Cloudflare API access and manual effort.

There are also two backup files committed in the repo directory which may leak
infra identifiers.

### Fix applied

Created `manifest/terraform/backend.tf.example` with two ready-to-use options:

- **Option A (Cloudflare R2)** — S3-compatible backend, free tier, zero egress
  cost, naturally co-located with the rest of the Cloudflare stack.
- **Option B (Terraform Cloud)** — free for 1 user, UI for state inspection.

Migration: rename to `backend.tf`, fill in credentials (store in `.env.enc`),
run `terraform init -reconfigure && terraform state push terraform.tfstate`.

## Finding 6 — Sentinel had no console UI integration (functionality gap)

| Field | Detail |
|-------|--------|
| **Severity** | Functionality |
| **Files** | Multiple — see below |
| **Status** | Fixed |

### What was wrong

The console UI had four apps (Identities, Keys, Certs, Provision) but no
Sentinel integration. Sentinel's `/v1/*` API existed with only a single
per-repository findings endpoint. There was no way to see installations,
repositories, or scan history from the console.

### Fix applied

**Sentinel service — new dashboard API** (`app/api/v1/dashboard.py`):

| Endpoint | Returns |
|----------|---------|
| `GET /v1/installations` | All GitHub App installations |
| `GET /v1/repositories` | All monitored repositories |
| `GET /v1/scans?limit=50&repository_id=N` | Recent scans (filterable) |

All three protected by `sentinel:read` scope via Mainframe token validation
(same auth pattern as the existing `/v1/repositories/{id}/findings`).

**Mainframe BFF — new sentinel proxy** (`app/api/v1/admin/sentinel.py`):

Routes `GET /admin/sentinel/{installations,repositories,scans,repositories/{id}/findings}`
to the Sentinel service using a `SENTINEL_JWT_TOKEN` service token. Superuser-gated
like all other BFF routes.

**Mainframe config** — two new settings:
```
SENTINEL_URL=http://szejo-control-plane-sentinel:8000
SENTINEL_JWT_TOKEN=<service token with sentinel:read scope>
```

**Console UI:**
- New `SentinelApp.tsx` component — 3-tab Windows 95-style panel:
  - **Installations** — GitHub account, install ID, suspended status
  - **Scans** — repo selector, recent scans table with status colour coding
  - **Findings** — repo + state selector, findings table with severity badges
    and summary counts (critical/high/medium/low/info)
- `api.ts` — added `SentinelInstallation`, `SentinelRepository`, `SentinelScan`,
  `SentinelFinding` types and four fetch functions.
- `Desktop.tsx` — registered `SentinelApp` in the `APPS` array (abbr: "SC",
  default size 860×540).

### Required manual step after deploy

Issue a Mainframe service token with `sentinel:read` scope and store it:
```bash
# From mainframe admin API (requires superuser JWT):
# POST /admin/services/tokens  { "client_id": "platform-svc", "scope": "sentinel:read" }
# OR via the services admin panel in Mainframe.
python3 -m scripts.szejo secrets set SENTINEL_JWT_TOKEN
```

Then add `SENTINEL_JWT_TOKEN` to `.env.template` and re-encrypt.
