## Finding 1 — `szejo secrets run` injected the entire pass store into every subprocess

| Field | Detail |
|-------|--------|
| **Severity** | Medium |
| **File** | `szejo_cli/secret_store.py` |
| **Status** | Fixed |

### What was wrong

`szejo secrets run -- <cmd>` decrypts the whole pass store (95 keys — every
Postgres password, `JANUS_UNSEAL_KEY` — the KMS master-key seed,
`GITHUB_APP_PEM`, `CLOUDFLARE_API_TOKEN`, ...) and merges all of it into the
wrapped subprocess's environment, unconditionally. `build_env()` had no
concept of scope:

```python
def build_env(tf: bool = False) -> dict[str, str]:
    data = env_crypto.load_dict()
    env = {**os.environ, **{k: v for k, v in data.items() if v}}
    ...
```

Every documented invocation went through this — including narrow,
single-purpose ones. `szejo secrets run -- szejo coder push szejo-atlas`
only needs `GITHUB_TOKEN` and Atlas's own six secrets, but its process (and
anything it or `coder templates push` shells out to) received all 95 keys,
including secrets for services with no relation to Atlas at all. Any supply-
chain compromise in a downstream tool invoked this way — a malicious
Terraform provider, a compromised `coder` CLI dependency, a bad npm postinstall
during a template push — would have had blanket access to every credential in
the platform, not just the ones relevant to the task at hand. This is a
straightforward violation of least privilege, and it's the kind of thing
that's easy to not notice because the tool "just works" — nothing *fails*
when scope is too broad.

### Fix applied

`build_env()` now detects the wrapped command's shape and scopes the merged
secrets to exactly what that shape needs:

| Command shape | Keys injected |
|---|---|
| `docker compose ...` | every `${VAR}` referenced across the 5 compose files (static grep, so it can't silently drift out of sync — 48 of 95 keys) |
| `terraform ...` (with `--tf`) | nothing beyond `TF_VAR_*` |
| `szejo coder push <template>` | `CODER_URL`, `CODER_SESSION_TOKEN`, plus that template's own variable map (already defined in `coder.py`'s `TEMPLATES` dict — reused, not duplicated) |
| `szejo certs ...` | `AEGIS_URL`, `JANUS_URL`, `AEGIS_MGMT_TOKEN` |
| `szejo bootstrap --emit-imports` | the 4 `CLOUDFLARE_*` variables |
| anything else | refused — must pass `--keys KEY1,KEY2` (explicit allowlist) or `--all` (loud, logged opt-out) |

Every real invocation documented in `README.md` and
`docs/runbooks/remote-access.md` matches one of the first five shapes, so no
existing documented workflow needed to change. Verified live: `docker compose
config -q` passes under the scoped env, `terraform plan` against production
still reports "No changes" under the narrowed TF-only scope, and a real
`szejo secrets run -- szejo coder push szejo-atlas` completed successfully
with none of sentinel's, portfolio's, or the control plane's own secrets in
its process environment.

### Remaining risk

The `docker compose` scope is still fairly broad (48 keys) because a plain
`docker compose up -d` genuinely starts every control-plane service at once,
each needing its own secrets. Scoping *that* further would mean resolving
which services a given `up -d`/`restart` call actually targets and
intersecting against a per-service key map — deferred; the coarser
compose-wide scope already closes the gap for every other command shape
(coder push, certs, terraform, bootstrap), which were the ones with no
principled reason to see unrelated secrets at all.
